## 1. 페이지 개요

### 1.1 페이지 목적

- 주최자와 참여자가 현재 이벤트의 기간, 참여자 목록, 가용 시간 현황을 확인한다.
- 공유 링크 복사와 참여/수정 CTA를 같은 상세 페이지 안에서 제공한다.
- 상세 화면은 `GET /api/events/{eventId}` 응답 하나만으로 다시 그릴 수 있어야 한다.

### 1.2 대상 사용자

| 사용자 유형 | 설명 |
| --- | --- |
| 1차 사용자 | 이벤트를 만든 주최자 |
| 2차 사용자 | 공유 링크로 들어온 참여자 |

### 1.3 완료 기준

- 사용자가 이벤트 기간과 참여자 현황을 오해 없이 이해할 수 있어야 한다.
- 프론트엔드는 `GET /api/events/{eventId}` 응답만으로 화면 상태를 다시 그릴 수 있어야 한다.
- 사용자가 상세 페이지 안에서 `참여/수정` 패널을 열 수 있어야 한다.
- 사용자가 현재 상세 페이지 링크를 복사할 수 있어야 한다.

## 2. 공통 제약 및 정책

| 항목 | 규칙 |
| --- | --- |
| 시간 기준 | API 입출력은 UTC 기준이다. |
| 화면 표시 | 사용자 현지 시간 표시 여부와 변환 책임은 프론트엔드가 가진다. |
| 슬롯 단위 | 30분 고정 |
| 데이터 기준 | 상세 화면은 항상 최신 `EventDetail` 응답을 기준으로 렌더링한다. |
| 참여자 목록 | API 응답의 `members`를 기준으로 렌더링한다. |
| 오류 처리 기준 | 프론트엔드는 `error.code`로 분기한다. |
| 404 처리 | 이벤트가 없거나 만료되면 `EVENT_NOT_FOUND` 상태 화면을 노출한다. |

## 3. 사용자 목표와 핵심 시나리오

| 시나리오 ID | 사용자 목표 | 시작 조건 | 완료 조건 | 우선순위 |
| --- | --- | --- | --- | --- |
| S-01 | 이벤트 현황을 보고 싶다 | `/events/{eventId}`로 진입했다 | 이벤트 기간과 참여자 현황을 확인한다 | `Must` |
| S-02 | 다른 사람에게 이벤트를 전달하고 싶다 | 상세 화면을 보고 있다 | 링크 복사 또는 공유 액션을 수행한다 | `Should` |
| S-03 | 내 일정을 등록/수정하고 싶다 | 상세 화면을 보고 있다 | 같은 페이지 안에서 참여/수정 패널을 연다 | `Must` |

## 4. 진입/이탈 경로

### 4.1 진입 경로

| 진입 방식 | 설명 |
| --- | --- |
| 생성 성공 후 이동 | `POST /api/events` 성공 뒤 리다이렉트 |
| 직접 URL 진입 | 공유받은 `/events/{eventId}` 링크로 접속 |

### 4.2 이탈 경로

| 이탈 목적지 | 조건 | 방식 |
| --- | --- | --- |
| 동일 페이지의 참여/수정 패널 | 사용자가 참여 CTA를 클릭 | 페이지 내 패널 열기 |
| 공유 액션 | 사용자가 공유 버튼을 클릭 | 링크 복사 또는 공유 시트 |

## 5. 정보 구조

### 5.1 화면 구성

| 영역 | 목적 | 포함 요소 | 우선순위 |
| --- | --- | --- | --- |
| 헤더 | 이벤트 정보와 핵심 액션 인지 | 이벤트 제목, 기간, 공유 버튼 | `High` |
| 주요 콘텐츠 | 전체 가능 시간 확인 | 시간표 또는 가용 현황 시각화, 참여자 목록 | `High` |
| 보조 정보 | 현재 맥락 이해 | 시간대 안내, 슬롯 단위 안내 | `Medium` |
| 참여/수정 진입 영역 | 다음 행동 유도 | `내 일정 등록/수정` CTA | `High` |

### 5.2 구현 상세 규칙

1. 페이지 진입 시 `GET /api/events/{eventId}`를 먼저 호출한다.
2. 조회 성공 시 `EventDetail` 전체 응답으로 헤더, 시간표, 참여자 목록을 렌더링한다.
3. 조회 실패가 `EVENT_NOT_FOUND`면 전용 빈 상태를 렌더링한다.
4. 사용자가 `내 일정 등록/수정`을 선택하면 같은 상세 페이지 안의 참여/수정 패널을 연다.
