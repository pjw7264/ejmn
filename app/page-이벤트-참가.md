## 문서 기준

- 이 문서는 `최종 목표 요구사항`과 `현재 구현 상태`를 함께 다루되, 현재 구현 상태를 우선 기준으로 본다.
- `PATCH`, 인증, 오류 분기 규칙은 최종 제품 요구사항으로 유지한다.
- 다만 현재 구현은 저장 제출 전 단계의 `입력 레퍼런스 패널`이므로, 미구현 항목은 현재 상태와 분리해서 읽어야 한다.

## 1. 페이지 개요

### 1.1 페이지 목적

- 주최자와 참여자가 자신의 이름, 선택 비밀번호, 가능 시간 입력 UI를 확인한다.
- 이 문서는 별도 URL 페이지가 아니라 `/?{eventId}` 상세 화면 안에서 열리는 참여/수정 패널의 요구사항을 다룬다.
- 현재 구현 기준으로 참여 패널은 상세 화면 안에서 열리는 입력 레퍼런스 패널이다.

### 1.2 대상 사용자

| 사용자 유형 | 설명 |
| --- | --- |
| 1차 사용자 | 이벤트를 막 생성하고 자신의 가능 시간을 입력하려는 주최자 |
| 2차 사용자 | 공유 링크로 들어와 기존 이벤트에 참여하려는 사용자 |
| 3차 사용자 | 같은 이름으로 기존 일정을 수정하려는 사용자 |

### 1.3 완료 기준

- 사용자가 이름, 비밀번호, 가능 시간 입력 레이아웃을 이해할 수 있어야 한다.
- 가능 시간 입력 UI는 이벤트 범위와 슬롯 단위를 기준으로 동작해야 한다.
- 최종 저장 연동 전에도 입력 구조와 패널 진입 흐름을 검토할 수 있어야 한다.

## 2. 공통 제약 및 정책

| 항목 | 규칙 |
| --- | --- |
| 시간 기준 | 입력은 사용자 현지 시간으로 보이더라도, API 전송은 UTC RRULE 규칙을 따라야 한다. |
| 참여자 식별자 | 참여자 수정 식별자는 `name` 이다. |
| 인증 방식 | `Authorization: Basic <base64(name:password)>` 를 사용한다. |
| 비밀번호 | 선택 입력이지만, 기존 참여자가 비밀번호를 가진 경우 수정 시 필요하다. |
| 이름 변경 | 기존 참여자의 이름 변경은 지원하지 않는다. |
| RRULE 제약 | 입력 결과는 `PATCH /api/events/{eventId}`가 허용하는 단일 RRULE 형식으로 직렬화 가능해야 한다. |
| 데이터 기준 | 패널은 먼저 `GET /api/events/{eventId}`로 이벤트 기간을 읽은 뒤 입력 범위를 제한한다. |
| 오류 처리 기준 | 입력 형식 오류와 인증 오류는 서버 응답 기준으로 표기한다. |

## 3. 사용자 목표와 핵심 시나리오

| 시나리오 ID | 사용자 목표 | 시작 조건 | 완료 조건 | 우선순위 |
| --- | --- | --- | --- | --- |
| S-01 | 내 일정 입력 구조를 확인하고 싶다 | 상세 페이지에서 참여 패널을 열었다 | 이름, 비밀번호, 가능 시간 입력 구성을 확인한다 | `Must` |
| S-02 | 가능 시간 선택 방식을 이해하고 싶다 | 패널을 열었다 | 가능 시간 입력 레퍼런스를 확인한다 | `Must` |
| S-03 | 저장 연동 전에 입력 범위를 검토하고 싶다 | 입력 UI를 보고 있다 | 이벤트 범위 안에서 동작하는지 확인한다 | `Should` |

## 5. 정보 구조

### 5.1 화면 구성

| 영역 | 목적 | 포함 요소 | 우선순위 |
| --- | --- | --- | --- |
| 헤더 | 현재 액션 인지 | 패널 제목 | `High` |
| 본문 | 이름과 가능 시간 입력/수정 | 이름 입력, 비밀번호 입력, 가능 시간 등록 컴포넌트 | `High` |
| 보조 정보 | 정책 안내 | 현재 입력 목적과 가능 시간 선택 컴포넌트 안내 | `Medium` |

### 5.2 구현 상세 규칙

1. 패널을 열기 전에 `GET /api/events/{eventId}` 결과가 있어야 한다.
2. 현재 패널 상단에는 이름, 비밀번호 입력과 가능 시간 입력 레퍼런스가 함께 렌더링된다.
3. 가능 시간 입력 컴포넌트는 이벤트의 `start`, `end`, `slotMinutes` 범위 안에서만 동작해야 한다.
4. 현재 구현에는 `저장하기` 버튼과 실제 `PATCH` 제출 경로가 없다.
5. 최종 `PATCH` 제출과 인증 오류 분기 규칙은 요구사항으로 유지하되, 현재 화면 구현은 입력 레퍼런스 단계다.

### 5.3 요구사항 구분

- `현재 구현`: 패널 열기/닫기, 이름 입력, 비밀번호 입력, 가능 시간 입력 레퍼런스 노출
- `목표 요구사항`: 저장 버튼, 실제 `PATCH` 제출, 인증 실패 분기, 성공 후 상세 화면 갱신
